# Drift between Zod env schema and `.env.example`; real secrets briefly landed in the public template

**Date:** 2026-04-22 22:44
**Files:** `.env.example`
**Severity:** high (security adjacent — near-miss secret leak)

## Problem

Two issues surfaced together while finalizing the `feature/nextjs-app` milestone:

1. **Schema drift.** `apps/web/lib/env.ts` and downstream code (`lib/supabase/browser.ts`, Sentry configs, future Sprint 2/3 integrations) reference env vars that were not present in the shared template `.env.example`:
   - `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`
   - `NEXT_PUBLIC_SENTRY_DSN`, `NEXT_PUBLIC_APP_URL`
   - `LOG_LEVEL`
   - `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`
   - `TRIGGER_API_KEY`, `TRIGGER_API_URL`

   A new dev cloning the repo would miss them entirely until a runtime error or a silent browser-client failure.

2. **Near-miss secret leak.** While trying to patch (1) manually, real Supabase `DATABASE_URL` password, `SUPABASE_ANON_KEY` and **`SUPABASE_SERVICE_ROLE_KEY`** were pasted into `.env.example` (a git-tracked file). The file was not committed or pushed, but `.env.example` is precisely the wrong place for real values — one `git add -A` + `git commit` would have published them.

## Root Cause

1. **Schema drift:** no automated link between Zod env schema and `.env.example`. Each new env reference added to `env.ts` (or to a feature file like `supabase/browser.ts`) has to be manually mirrored into the template. There is no test/CI check that fails when they disagree.
2. **Secret-in-template:** conceptual confusion — `.env.example` vs `.env` / `.env.local`. Next.js convention: `.env.local` (gitignored) holds real values; `.env.example` (tracked) is a placeholders-only template. Here both copy-targets were `.env*`-like and both looked like "the place where env goes", so real values flowed into the wrong one. Additionally, the original `.env.example` already used partially-real-looking strings (`postgres.xxx:password@...pooler.supabase.com`), which lowered the mental guard against putting real values next to them.

## Solution

1. Restored `.env.example` to **placeholders only** (with explanatory comments) and **added all missing variables** referenced by `env.ts` / new code paths. Each block tagged with its sprint (`Sprint 2`, `Sprint 3`) so optional vars don't look mandatory.
2. Added a header comment in `.env.example` pointing to `apps/web/.env.local` as the real-values target, plus an explicit reminder that secrets are never committed and a reference to `docs/env-vars.md`.
3. Explicitly marked `SUPABASE_SERVICE_ROLE_KEY` as `# SERVER ONLY — обходит RLS` to raise the mental guard.
4. Verified smoke: `pnpm typecheck` + `pnpm build` still green after the template change (template is not consumed at build time, but the check confirms no regression).

No runtime code was changed — this was a configuration/documentation fix. Real secrets were moved by the user to a gitignored `.env` file; Supabase keys were NOT rotated (user stated the prior paste was never exposed outside this local session; assessed as acceptable near-miss, not an active breach).

## Prevention

1. **Schema/template drift check** — when Sprint 2 CI is set up, add a small script that parses `apps/web/lib/env.ts` Zod keys and greps them in `.env.example`; fail CI on mismatch. Mental model: "if it's in the schema, it must be in the template."
2. **Never put real values anywhere a `git diff` shows them** — real values live only in `.env.local` / `.env` (gitignored) or in Vercel env dashboard. `.env.example` is a public artifact by construction.
3. **Review any `.env.example` edit** like you would review a code change for secrets — it's the single file in the repo specifically designed to look like it accepts env values.
4. **Naming hygiene in the template** — use obvious placeholders like `xxx`, `password`, `changeme`, never strings that look real enough to be confused with a working value.
5. Code review checklist: when adding a new `process.env.FOO` or a new `z.object({ FOO: ... })` branch in `env.ts`, verify `.env.example` was updated in the same PR.
