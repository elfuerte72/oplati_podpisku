# Глоссарий

## Продуктовые термины

**Заказ (Order)** — запись в БД, представляющая намерение клиента оплатить конкретную подписку. Имеет состояние (статус), сумму, назначенного оператора.

**Черновик (Draft)** — заказ в статусе `draft`/`clarifying`/`ready_for_payment`, ещё не оплаченный.

**Каталог (Catalog)** — справочник известных сервисов (`services`) с тарифами. Заказы могут быть **каталожными** (указан `service_id`) или **custom** (`custom_service_description`).

**KYC (Know Your Customer)** — проверка личности клиента. Требуется для некоторых сервисов (Airbnb, LinkedIn). Флаг `services.requires_kyc`.

**Handoff (Эскалация)** — переключение диалога с AI на живого оператора. Реализуется через forum-topic в Telegram-группе.

**Fulfillment (Выполнение)** — процесс, когда оператор фактически оплачивает подписку у иностранного сервиса за клиента.

**Provisioning** — то же, что fulfillment; применяется чаще в контексте «выдать клиенту доступ».

**Short ID** — человекочитаемый идентификатор заказа (`ORD-7KX42`), в отличие от внутреннего UUID.

## Роли

**User** — клиент, покупатель подписки. Живёт в `users`.

**Operator** — сотрудник, обрабатывающий заказы. Staff role.

**Supervisor** — лид смены. Видит всё, одобряет возвраты.

**Admin** — владелец/администратор системы.

**Staff** — общий термин для operator/supervisor/admin.

## Технические термины

**ACID** — свойства транзакций БД: Atomicity, Consistency, Isolation, Durability. Гарантируют, что операция либо выполнится целиком, либо не выполнится. Критично для финансовых данных.

**RLS (Row Level Security)** — правила доступа к данным на уровне строк таблицы, встроенные в PostgreSQL. В Supabase — основа защиты пользовательских данных.

**Realtime** — подписка на изменения в БД через WebSocket, push от сервера к клиенту. В Supabase — на базе логической репликации.

**Auth (Authentication + Authorization)** — идентификация «кто» и проверка прав «что можно». Для клиентов идентичность = `telegram_id` или `web_session_id`; для staff — Supabase Auth JWT.

**Webhook** — HTTP POST от внешнего сервиса к нашему endpoint при событии (платёж, Telegram-сообщение).

**Idempotency (Идемпотентность)** — свойство операции давать тот же результат при повторном вызове. Для webhook'ов гарантируется через `UNIQUE(provider, provider_ref)`.

**Fluid Compute** — runtime Vercel, переиспользующий function instances, снижающий cold start. Заменяет классический «один request на instance» serverless.

**Modular Monolith** — архитектурный стиль: единый deploy, но чёткие внутренние границы (через пакеты). Позднее можно разделить на сервисы без переписывания.

**Turborepo** — build system для monorepo, инкрементальные сборки, кэш.

**pgBouncer / Pooler** — connection pooler перед PostgreSQL. Supabase использует в «Transaction mode» на порту 6543. Не поддерживает prepared statements → `prepare: false` в postgres-js.

**HMAC** — Hash-based Message Authentication Code. Способ проверить подлинность и целостность сообщения на основе общего секрета.

**Signed URL** — временная ссылка на файл в Storage, проверяемая сервером без передачи секретов клиенту. TTL обычно 15 минут – 1 час.

## AI / Claude

**System prompt** — инструкция для модели, задающая её роль и правила. Наш — см. `ai-agent.md`.

**Tool use (Tool calling)** — возможность модели вызывать заранее описанные функции (search_catalog, propose_order и т.п.) с валидируемым JSON input.

**Agentic loop** — цикл «модель → tool use → результат → модель → …» до `end_turn`. Наш лимит — 5 итераций.

**Context window** — максимум токенов истории, которые мы подаём модели. У Claude Opus 4.6 — сотни тысяч, мы используем ~10k.

**Prompt injection** — атака, когда пользователь пытается перезаписать system prompt через своё сообщение. Защита — фильтрация + явный запрет в system prompt.

## Платежи

**Invoice (Инвойс)** — запись о намерении получить оплату с клиента. В нашей БД — `payments` с `status='pending'`.

**Provider ref** — идентификатор платежа у провайдера (YooKassa payment id, CryptoBot invoice_id). Основа идемпотентности.

**Reconciliation** — сверка состояния платежей между нами и провайдером. Cron-задача `poll-payment` делает это.

**Chargeback** — возврат платежа по инициативе держателя карты через банк. Для нас негативный исход, нужен супервизор.

## Единицы

**Копейка** — 1/100 рубля. В БД все суммы хранятся в копейках как `integer`.

**Сент** — 1/100 единицы иностранной валюты. `original_amount` в минимальных единицах.

## Процессы

**Sprint** — 1–2 недели фокусной работы с чёткой целью. Sprint 1 = MVP скелет; Sprint 2 = заказы + платежи; Sprint 3 = веб-чат + prod-ready.

**Definition of Done (DoD)** — чеклист критериев завершения задачи/спринта.

**Runbook** — инструкция «что делать при X». Мы ведём для операторов и для инцидентов.

**Postmortem** — разбор инцидента после разрешения. Цель — учиться, не наказывать.

## Инфраструктурные сущности

**Environment** — среда развёртывания: `development` (локально), `preview` (PR previews на Vercel), `production` (prod).

**Feature flag** — переключатель для включения/выключения фичи в проде без деплоя. На MVP не используем, оставлено в backlog.

**Migration** — SQL-скрипт изменения схемы БД. Мы используем Drizzle migrations, forward-only.

**PITR (Point-in-Time Recovery)** — возможность восстановить БД на любой момент в прошлом (обычно до 7 дней). Pro-фича Supabase.
