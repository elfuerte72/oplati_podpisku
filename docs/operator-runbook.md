# Operator Runbook

Документ для команды операторов и супервизоров. Описывает ежедневные процессы.

## Роли

### Operator
- Обрабатывает назначенные заказы
- Общается с клиентами через Telegram forum-topics
- Подтверждает выполнение заказов скриншотами

### Supervisor
- Видит все заказы
- Переназначает при необходимости
- Одобряет возвраты
- Решает споры
- Первый эскалационный уровень

### Admin (владелец)
- Каталог, цены
- Добавление/отключение операторов
- Настройки

## Начало смены

1. Открыть https://oplati.example.com/admin (логин через email + пароль)
2. Проверить дашборд: количество `paid` без назначенного оператора
3. Взять в работу те, что в очереди (кнопка «Взять» на карточке)

Автоназначения на MVP нет — операторы берут сами.

## Обработка заказа

### Шаг 1: Получить уведомление

При оплате заказа появляется:
- Запись `paid` в `/admin/orders` (с realtime)
- Topic в TG-группе операторов с кратким снапшотом

### Шаг 2: Взять в работу

В админке на карточке заказа — кнопка «Взять в работу».
После этого:
- `status = in_fulfillment`
- `assigned_operator_id = ты`
- Клиент получает сообщение: «Заказ в работе»
- Topic становится твоим (другие операторы не вмешиваются без `/take`)

### Шаг 3: Оплатить у иностранного сервиса

**Как именно** — определяется твоей командой (виртуальная карта / крипта / чужой аккаунт). Этот флоу вне бота.

### Шаг 4: Подтвердить выполнение

В админке → карточка заказа → «Подтвердить»:
1. Загрузить скриншот (квитанция / письмо о подтверждении подписки / скрин личного кабинета сервиса) — `kind = fulfillment_proof`
2. Опционально текст клиенту: «Подписка активна, проверьте почту»
3. Кнопка «Готово» → `status = completed`

Клиент автоматически получает уведомление с скриншотом.

### Шаг 5: Если не получилось

Не удалось оплатить (карта отклонена, аккаунт заблокирован, сервис недоступен):
1. Нажать «Не удалось» → ввести причину (только для внутреннего ревью, клиенту не показывается)
2. `status = failed`
3. Клиент получает сообщение: «К сожалению, не получилось. Свяжемся для возврата».
4. Эскалация супервизору на refund

## Общение с клиентом в TG-topic

### Правила
- **На «вы»**, если клиент сам не перешёл на «ты»
- Представляться один раз в начале: «Это Алексей, оператор поддержки»
- **Без маркетингового языка** — коротко, по делу
- Отвечать в течение 5 минут (SLA) в рабочее время оператора
- При долгих операциях — предупредить: «Подождите 10 минут, оплачиваю»

### Команды в topic

| Команда | Что делает |
|---|---|
| `/take` | Назначить себя на заказ (если автонаправления не было) |
| `/note <текст>` | Приватная заметка, не отправляется клиенту |
| `/order ORD-XXX` | Показать карточку другого заказа клиента |
| `/ai_back` | Вернуть диалог AI — закрывает topic, бот снова отвечает |
| `/supervisor` | Эскалация супервизору (создаёт метку в админке) |

### Запрещено

- Обещать сроки, которые не можешь гарантировать
- Делиться внутренней информацией (цены закупки, имена других операторов)
- Принимать оплату вне официального флоу (прямой перевод и т.п.) — всегда через бот и провайдеры
- Просить у клиента пароль, 2FA-код, номер карты
- Оскорблять, спорить — эскалация супервизору

## Эскалация

Супервизору:
- Клиент угрожает чарджбеком
- Сервис заблокировал аккаунт клиента
- Конфликт между операторами
- Сумма > 10 000 ₽ и возникли сомнения
- Необычный запрос не из каталога

Команда: `/supervisor` в topic или пинг в отдельном чате.

## Возвраты

Initiate:
- Клиент просит — создать `refund_requested` (кнопка в админке)
- Оператор решил `failed` → автоматически `refund_requested`

Supervisor:
1. Проверяет legitimacy (не мошенничество)
2. Нажимает «Вернуть» в админке
3. Система возвращает деньги через провайдера (YooKassa refund / CryptoBot)
4. `status = refunded`

## Каталог

Если клиент запрашивает сервис не из каталога:
1. Продолжать диалог — AI может принять как custom-заказ
2. Оценить сложность
3. Если возможно — предложить цену (правило: закупочная цена × 1.15 + минимум 200 ₽)
4. Обработать как обычный заказ, `service_id = NULL`, `custom_service_description = подробно`

## KYC-кейсы

Сервисы с KYC (Airbnb, LinkedIn Premium, etc.):
1. AI обычно запрашивает базовые данные
2. Оператор проверяет и дооформляет: может потребоваться копия паспорта, адрес, IP-регион
3. Загружать документы через админку в bucket `kyc-documents` (не в чат Telegram)
4. После KYC — стандартный флоу

## Смены

- 24/7 покрытие — команда организует графики самостоятельно
- SLA первого ответа: 5 мин в пиковое время
- SLA fulfillment: 2 часа p95 (Критические сервисы — быстрее)

## Handover между сменами

Перед концом смены:
1. Убедиться, что все `in_fulfillment` твои — либо завершены, либо передать следующему
2. Написать в admin group: «Ухожу, у меня 2 активных заказа ORD-XXX, ORD-YYY, передал @next_operator»
3. Next operator — `/take` в соответствующих topics

## Что делать при проблемах

| Проблема | Действие |
|---|---|
| Админка не открывается | Проверить interner, browser cache; написать admin |
| Бот в TG не отвечает | Смотреть в #status-канал admin group |
| Клиент хамит | Ответить вежливо, эскалация через `/supervisor` |
| Платёж прошёл, но статус не обновился | Проверить запись в payments таблицу (админ может); ручной перевод через supervisor |
| Скриншот не загружается | Проверить размер < 10 MB и формат (PNG/JPG/PDF) |
